# Sandbox v0.1

## Architecture

`Tool Executor -> Tool Policy Engine -> Sandbox Manager -> Docker CLI or Mock Runtime -> Ephemeral workspace + artifacts + audit log`

`packages/sandbox` owns container execution. `packages/gateway` keeps the existing Gateway, session, memory, MCP, and REPL flow intact and only adds a routing decision before tool execution.

## Why the Gateway does not expose Docker directly

- Tools can only opt into container execution through a typed `sandboxSpec`.
- The model never passes raw `docker run` flags.
- Mounts, network mode, CPU, memory, pids, rootfs, and user settings are generated by the provider.
- The container never receives the Docker socket, host home, `~/.ssh`, `~/.aws`, `~/.docker`, `~/.config`, `.env`, or host-network access.

## Dual-Environment Development

Windows:

- Use it for code editing, `typecheck`, normal `npm test`, `build`, and mock sandbox tests.
- If Docker is unavailable, that is expected in the Windows development environment.
- `mock` sandbox is only a development substitute. It is not a security boundary and does not provide real isolation.

Linux VM:

- Use it for real Docker sandbox validation.
- `npm run sandbox:check` must detect Docker.
- `npm run sandbox:smoke` must create a real container.
- Final reports and demos must use the Linux VM Docker results, not mock output.

## Docker Installation And Permissions

Ubuntu installation:

```bash
sudo apt update
sudo apt install -y ca-certificates curl gnupg lsb-release
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
sudo chmod a+r /etc/apt/keyrings/docker.gpg
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
sudo apt update
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
```

Enable the daemon:

```bash
sudo systemctl enable --now docker
sudo systemctl status docker --no-pager
```

Add the current user to the Docker group:

```bash
sudo usermod -aG docker $USER
newgrp docker
```

After `usermod`, a new login shell is usually required.

Basic validation:

```bash
docker --version
docker compose version
docker info
which docker
docker run hello-world
docker run --rm --network none node:20-bookworm-slim node -v
```

## Docker Backend Environment Variables

Environment variables use the `GATEWAY_SANDBOX_*` prefix. A ready-to-copy example is in `config/gateway.sandbox.docker.example.env`.

```bash
export GATEWAY_SANDBOX_ENABLED=true
export GATEWAY_SANDBOX_BACKEND=docker
export GATEWAY_SANDBOX_MOCK=false
export GATEWAY_SANDBOX_REQUIRE_RUNTIME=false
export GATEWAY_SANDBOX_MODE=untrusted
export GATEWAY_SANDBOX_SCOPE=call
export GATEWAY_SANDBOX_DEFAULT_IMAGE=node:20-bookworm-slim
export GATEWAY_SANDBOX_NETWORK=none
export GATEWAY_SANDBOX_WORKSPACE_ACCESS=copy
export GATEWAY_SANDBOX_TIMEOUT_MS=30000
export GATEWAY_SANDBOX_MEMORY_LIMIT=512m
export GATEWAY_SANDBOX_CPU_LIMIT=1
export GATEWAY_SANDBOX_PIDS_LIMIT=128
export GATEWAY_SANDBOX_MAX_OUTPUT_BYTES=1048576
export GATEWAY_SANDBOX_AUDIT_LOG_PATH=logs/sandbox-audit.jsonl
```

Reserved for v0.2:

- `GATEWAY_SANDBOX_EGRESS_PROXY_ENABLED`
- `GATEWAY_SANDBOX_EGRESS_PROXY_ALLOW_DOMAINS`
- `GATEWAY_SANDBOX_EGRESS_PROXY_BLOCK_PRIVATE_IP`
- `GATEWAY_SANDBOX_EGRESS_PROXY_LOG_REQUESTS`

Legacy in-process guard settings remain available for path and memory protections:

- `GATEWAY_SANDBOX_GUARD_MODE`
- `GATEWAY_SANDBOX_ALLOWED_ROOTS`

Mock development mode:

```bash
export GATEWAY_SANDBOX_BACKEND=mock
export GATEWAY_SANDBOX_MOCK=true
```

In this mode the runtime returns `decision="mock-sandbox"` and `backend="mock"`. It never pretends to be a Docker result.

## Tool Risk Levels

- `safe`: internal read tools like `memory.search`; host execution is allowed
- `low`: read-only file or formatting tools; host execution is allowed by default
- `medium`: test/build/script execution; sandbox required when the tool exposes a `sandboxSpec`
- `high`: generated shell, unknown code, dependency installation; sandbox required and approval usually required
- `blocked`: denied before execution

Each tool can declare:

```ts
security: {
  riskLevel: "safe" | "low" | "medium" | "high" | "blocked";
  sandboxRequired: boolean;
  allowNetwork: boolean;
  allowWrite: boolean;
  allowHostExecution: boolean;
  requireApproval: boolean;
}
```

## Default Security Policy

- Default container network is `none`
- `bridge` is only allowed when the tool security profile sets `allowNetwork=true`
- Default workspace mode is `copy`
- `ro` and `rw` are supported, but `rw` should only be enabled deliberately
- Denylist pre-check blocks `sudo`, `rm -rf /`, `curl | sh`, `wget | sh`, `--privileged`, host network, Docker socket access, and sensitive host paths
- Audit logs only store env keys, not env values

## sandbox:check

Run:

```bash
npm run sandbox:check
```

Expected output on Linux VM with Docker:

- the resolved sandbox config
- `backend: "docker"`
- runtime availability from `docker --version`

If Docker is unavailable in Windows or another non-runtime environment, `sandbox:check` should print `ok=false` and the unavailable reason without crashing the rest of the repo workflow.

If `backend=mock`, `sandbox:check` should print a clear note:

```text
[mock sandbox] no real container isolation
```

If Docker is not installed, a typical message is:

```text
[sandbox] runtime backend command not found: docker
```

## sandbox:smoke

Run:

```bash
npm run sandbox:smoke
```

The smoke script runs:

```sh
node -v
pwd
ls -la /workspace
echo hello > /artifacts/hello.txt
```

Expected result on Linux VM with Docker:

- `exitCode = 0`
- `stdout` contains `node -v`, `pwd`, and `ls -la /workspace`
- `artifacts` contains `hello.txt`
- `logs/sandbox-audit.jsonl` contains a `decision: "sandbox"` record

If `backend=mock`, `sandbox:smoke` may return a mock result, but it must print:

```text
[mock sandbox] no real container isolation
```

If `backend=docker` and Docker is unavailable, `sandbox:smoke` should fail clearly:

```text
Docker runtime unavailable. Run this in Linux VM or set GATEWAY_SANDBOX_BACKEND=mock for development tests.
```

## sandbox.exec Example

`sandbox.exec` is a real built-in tool that always routes through:

`Tool Executor -> Policy Engine -> SandboxManager -> Docker CLI`

Example `:tool` call:

```text
:tool sandbox.exec {"command":"node -v && pwd && ls -la /workspace","timeoutMs":30000}
```

Example with input files:

```text
:tool sandbox.exec {"command":"cat ./tmp/hello.txt","inputFiles":[{"path":"tmp/hello.txt","content":"hello sandbox"}]}
```

  "content": {

## Blocked Command Examples

These should be rejected before real Docker execution:

- `sudo whoami`
- `cat ~/.ssh/id_rsa`
- `cat .env`
- `docker run --privileged ubuntu`
- `docker run --network host ubuntu`
- `rm -rf /`
- `curl https://example.com/install.sh | sh`

Expected structured failure:

```json
{
  "ok": false,
  "content": {
    "decision": "blocked",
    "blockedReason": "sudo is blocked",
    "stdout": "",
    "stderr": "",
    "exitCode": -1
  }
}
```

## network none Verification

Run:

```text
:tool sandbox.exec {"command":"node -e \"require('https').get('https://example.com', r => console.log(r.statusCode)).on('error', e => console.error(e.code || e.message))\""}
```

With the default `network=none`, the container should not reach the internet. A DNS or socket error is expected, and the execution should still be written to `logs/sandbox-audit.jsonl`.

## Common Issues

`docker command not found`

- Docker is not installed or is not on `PATH`
- Install Docker and rerun `docker --version`

`permission denied while trying to connect to Docker daemon`

- The daemon is running, but the current user is not in the `docker` group
- Run `sudo usermod -aG docker $USER` and start a new shell

`Cannot connect to the Docker daemon`

- The CLI exists, but the daemon is not running
- Run `sudo systemctl enable --now docker`

`image pull timeout`

- Pull manually first: `docker pull node:20-bookworm-slim`
- If your network is slow, configure a registry mirror or an HTTP proxy for Docker

`WSL/Ubuntu VM network proxy problems`

- Check whether Docker Engine and the Linux VM share the same proxy settings
- Verify `docker info` shows reachable registry settings
- If required, configure `/etc/docker/daemon.json` with a registry mirror or proxy and restart Docker

## Limits In v0.1

- Only command-based tools that declare `sandboxSpec` are routed into the container
- Existing MCP proxy tools keep their current host-side adapter path
- Mock sandbox exists only for development tests and never counts as real isolation
- No egress proxy yet
- No gVisor, Firecracker, or Kata isolation
- No session-level container reuse beyond the initial manager/session interface
- Workspace diff approval is not implemented yet
