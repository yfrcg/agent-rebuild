# Sandbox v0.1

## Sandbox v0.1 Overview

AgentRebuild needs a sandbox because the Gateway can route model-selected tool calls into code execution paths that should not run directly on the host by default.

Architecture:

```text
Gateway / Tool Executor
  -> Tool Policy Engine
  -> SandboxManager
  -> Docker Runtime or Mock Runtime
  -> workspace / artifacts / audit
```

`packages/sandbox` owns the execution boundary. `packages/gateway` only adds the minimum policy and routing hooks required to decide whether a tool runs on the host, is blocked, or enters the sandbox path.

## Core Design

### SandboxManager

- Creates per-call or per-session sandbox directories
- Prepares workspace mounts or copied workspaces
- Selects the configured runtime provider
- Executes sandbox requests
- Collects artifacts
- Writes sandbox audit events

### Tool Policy Engine

- Maps tool security declarations to execution decisions
- Enforces `ToolSecurityProfile`
- Blocks known-dangerous commands before runtime execution
- Keeps blocked commands from reaching Docker or mock execution

### Runtime Provider

- `docker`: real Linux VM validation path
- `mock`: development-only substitute for Windows or non-Docker environments
- Docker CLI flags are generated by the sandbox layer, not by LLM input

### Workspace Strategy

- Supports `none`, `copy`, `ro`, `rw`
- Default is `copy`
- `copy` excludes `.env`, `.git`, `node_modules`, `dist`, `build`, `logs`, `.agent-rebuild`, `.ssh`, `.aws`, `.docker`, and similar sensitive or noisy directories

### Audit Log

- JSONL output at `logs/sandbox-audit.jsonl` by default
- Records decision, backend, image, command, mounts, limits, timing, and artifacts
- Records env keys only, never full env values

### Mock Backend

- Development substitute only
- Does not call Docker or Podman
- Does not execute real shell commands
- Must return `decision="mock-sandbox"` and `backend="mock"`
- Is not a security boundary

### Docker Backend

- Real container isolation path for Linux VM validation
- Must return `decision="sandbox"` and `backend="docker"`
- Must pass `sandbox:check` and `sandbox:smoke` in the Linux Docker environment

## Security Boundary

- Default `network=none`
- Default `workspaceAccess=copy`
- Real project directories are not `rw` mounted by default
- Host home is not mounted
- `~/.ssh` is not mounted
- `~/.aws` is not mounted
- `~/.docker` is not mounted
- `.env` is not mounted
- `/var/run/docker.sock` is not mounted
- `--privileged` is forbidden
- `--network host` is forbidden
- Docker parameters are generated by `SandboxManager` and runtime providers; the LLM cannot directly control Docker CLI flags
- Audit logs record env keys only, not env values
- Blocked commands are rejected before runtime execution

## Dual-Environment Strategy

### Windows

- Used for code editing, `typecheck`, normal `npm test`, `build`, and mock sandbox tests
- Windows does not pretend to have a real Docker sandbox when Docker is unavailable
- `mock` backend does not provide real security isolation
- `mock` backend must return `decision="mock-sandbox"`

### Linux VM

- Used for real Docker acceptance
- `docker` backend must return `decision="sandbox"`
- `sandbox:check` and `sandbox:smoke` are the final validation commands
- Final reports and demos must rely on Linux VM Docker results, not mock results

## Configuration

Sandbox environment variables:

- `GATEWAY_SANDBOX_ENABLED`
- `GATEWAY_SANDBOX_BACKEND`
- `GATEWAY_SANDBOX_MOCK`
- `GATEWAY_SANDBOX_REQUIRE_RUNTIME`
- `GATEWAY_SANDBOX_MODE`
- `GATEWAY_SANDBOX_SCOPE`
- `GATEWAY_SANDBOX_DEFAULT_IMAGE`
- `GATEWAY_SANDBOX_NETWORK`
- `GATEWAY_SANDBOX_WORKSPACE_ACCESS`
- `GATEWAY_SANDBOX_TIMEOUT_MS`
- `GATEWAY_SANDBOX_MEMORY_LIMIT`
- `GATEWAY_SANDBOX_CPU_LIMIT`
- `GATEWAY_SANDBOX_PIDS_LIMIT`
- `GATEWAY_SANDBOX_MAX_OUTPUT_BYTES`
- `GATEWAY_SANDBOX_AUDIT_LOG_PATH`

Default development template:

```bash
export GATEWAY_SANDBOX_ENABLED=true
export GATEWAY_SANDBOX_BACKEND=docker
export GATEWAY_SANDBOX_MOCK=false
export GATEWAY_SANDBOX_REQUIRE_RUNTIME=false
export GATEWAY_SANDBOX_MODE=untrusted
export GATEWAY_SANDBOX_SCOPE=call
export GATEWAY_SANDBOX_DEFAULT_IMAGE=node:20-bookworm-slim
export GATEWAY_SANDBOX_NETWORK=none
export GATEWAY_SANDBOX_WORKSPACE_ACCESS=copy
export GATEWAY_SANDBOX_TIMEOUT_MS=30000
export GATEWAY_SANDBOX_MEMORY_LIMIT=512m
export GATEWAY_SANDBOX_CPU_LIMIT=1
export GATEWAY_SANDBOX_PIDS_LIMIT=128
export GATEWAY_SANDBOX_MAX_OUTPUT_BYTES=1048576
export GATEWAY_SANDBOX_AUDIT_LOG_PATH=logs/sandbox-audit.jsonl
```

## Acceptance Commands

General project validation:

```bash
npm run typecheck
npm test
npm run build
```

Docker backend validation:

```bash
export GATEWAY_SANDBOX_BACKEND=docker
export GATEWAY_SANDBOX_MOCK=false
export GATEWAY_SANDBOX_REQUIRE_RUNTIME=true
export GATEWAY_SANDBOX_NETWORK=none
export GATEWAY_SANDBOX_WORKSPACE_ACCESS=copy

npm run sandbox:check
npm run sandbox:smoke
```

Mock backend validation:

```bash
export GATEWAY_SANDBOX_BACKEND=mock
export GATEWAY_SANDBOX_MOCK=true

npm run sandbox:check
npm run sandbox:smoke
```

## Expected Results

Docker backend:

- `decision="sandbox"`
- `backend="docker"`
- `ok=true`
- `exitCode=0`
- `artifacts` includes `hello.txt`
- audit log includes a real sandbox event

Mock backend:

- `decision="mock-sandbox"`
- `backend="mock"`
- output includes `no real container isolation`
- does not call Docker or Podman

## sandbox.exec Security Profile

The built-in `sandbox.exec` tool is declared with:

```ts
security: {
  riskLevel: "medium",
  sandboxRequired: true,
  allowNetwork: false,
  allowWrite: true,
  allowHostExecution: false,
  requireApproval: false
}
```

This is the intended v0.1 security contract and is already implemented in `packages/gateway/builtinTools.ts`.
