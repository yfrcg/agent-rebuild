Claude Code · Permissions

🔒权限系统 — 6种权限模式

从严格限制到完全放开的灵活安全梯度

🔒 Default 默认模式

所有写操作和Bash命令均需用户确认。最安全的默认行为，适合初次使用。

📝 acceptEdits 编辑模式

自动接受文件编辑操作，Bash仍需确认。适合代码重构等大量编辑场景。

📚 Plan 规划模式

只允许只读操作，禁止所有写入和执行。适合代码审查和分析任务。

⚡ Auto 自动模式

自动接受大部分操作，仅高危命令(如rm -rf)需确认。适合信任场景。

🔊 dontAsk 静默模式

不询问用户，受安全策略限制。适合自动化脚本和CI/CD集成。

🚨 bypass 绕过模式

跳过所有权限检查。仅限开发调试，需要显式flag启用。

📈 五层压缩体系

层级触发条件策略
L1 Micro单条消息过长截断工具输出，保留摘要
L2 Auto上下文达87%阈值压缩早期对话，保留最近N轮

L4 Cache-Aware缓存边界优化"外科手术式删除"保留缓存命中区

  • 双重控制：单次budget + 累计session budget
  • 超限策略：触发L2/L3压缩 → 释放空间 → 继续执行

📝 9段式完整摘要（Full Compaction）

  • ① 当前目标 — 用户正在做什么
  • ② 已完成工作 — 完成了哪些步骤
  • ③ 当前文件状态 — 涉及文件及状态
  • ④ 待办事项 — 还需做什么
  • ⑤ 关键决策 — 重要技术决策

  • ⑦ 环境信息 — 目录、分支、环境变量
  • ⑧ 用户偏好 — 会话中表达的偏好
  • ⑨ 下一步行动 — 压缩后应继续的步骤

🔄 Cache-Aware 压缩策略

  • 利用Anthropic API的 prompt caching 机制
  • system prompt和早期对话被缓存
  • "外科手术式删除"（surgical deletion）
  • 保留已缓存部分不动（避免cache invalidation）
  • 只删除缓存边界之后的内容
  • 最大化 cache hit rate，节省~80%成本

📄 getSystemPrompt() 动态组装

  • 静态部分（"System Constitution"）
    核心行为准则 · 工具使用指南 · 安全策略 · 代码风格规范

  • 在system prompt中明确标记缓存边界
  • 边界前内容被缓存，边界后不缓存
  • 类似 <!-- CACHE_BOUNDARY --> 文本标记
  • 确保静态内容稳定缓存、动态内容实时更新

  • 支持按用户/群组/通道粒度划分Agent会话和记忆
  • 通道间无缝切换，SOUL.md和MEMORY.md保持一致
设计优势：用户可以在电脑上用CLI开始任务，然后在手机上通过Telegram继续对话。Channel Bridges确保无论通过哪个渠道接入，智能体都保持一致的"人格"和记忆——实现了多智能体隔离和协作。

OpenClaw · Security

🔒安全机制与攻击面分析

Fail-Closed 设计 · 真实系统权限 · 复杂攻击面 · 安全研究重要案例

🛡 防御机制

  • Fail-Closed — 权限检查默认拒绝，不确定时宁可拒绝
  • 输入验证 — Schema校验 + 注入检测 + 频率限制
  • 审计日志 — 所有工具调用记录完整审计链，不可篡改
  • 技能隔离 — 每个技能运行在独立上下文
  • 显式授权 — 外部API调用需用户确认

⚠ 攻击面分析

  • 文件系统 — Agent可读写文件，存在数据泄露风险
  • Shell执行 — 可执行命令，存在远程代码执行风险
  • 浏览器控制 — 可操控浏览器，扩大攻击面
  • 网络访问 — 可发起HTTP请求，存在数据外传风险
  • 长期记忆 — 记忆文件可能包含敏感信息

🔒 安全研究价值

  • 提示注入 — 恶意指令隐藏在网页/邮件/文档中
  • 工具链攻击 — 多步工具调用组合绕过单点检测
  • 供应链污染 — 恶意技能伪装正常工具窃取数据
  • 记忆泄露 — API密钥、隐私数据可能被提取
  • 多Agent失控 — Agent互相调用导致任务失控
研究意义：OpenClaw具备真实的系统权限和复杂的攻击面，成为研究智能体安全、供应链攻击、提示注入、工具滥用和权限治理的重要案例。Agent越接近操作系统，安全治理就越重要。

⚠OpenClaw 面临的挑战与风险

🛡 安全风险

  • 权限配置不当 — Gateway暴露不安全或技能恶意行为
  • 远程代码执行 — 文件/Shell/浏览器/网络强工具风险

  • 恶意指令 — 隐藏在网页/邮件/文档/工具输出中
  • 顺序工具链 — 多步工具调用组合绕过单点检测
  • 记忆篡改 — 诱导Agent修改记忆文件植入后门

  • 恶意技能 — 伪装正常工具窃取API Key和隐私数据
  • 记忆泄露 — Markdown/JSONL/SQLite缺少加密和脱敏

  • 安全策略遗忘 — 摘要覆盖或遗忘安全约束

🤖 多Agent调度复杂

  • 任务路由 · 上下文隔离 · 并发控制 · 成本控制
  • Agent互相调用 — 任务失控、成本暴涨
  • 状态混乱 — 缺乏调度策略导致结果不确定
  • 使用门槛高 · 技能生态质量参差不齐

Part III · Philosophy

🧠设计哲学差异分析

两种截然不同的AI Agent设计思路

🎯 Claude Code 哲学

  • "精锐部队" — 少量但高度优化的工具
  • "深度优先" — 在编码场景做到极致
  • "平台主导" — 最佳实践由平台决定
  • "安全优先" — OS级沙箱严格限制
  • "单一模型" — 深度绑定Claude系列

核心理念：在特定领域做到最好，而非什么都做

🌱 OpenClaw 哲学

  • "人民战争" — 海量社区技能覆盖一切
  • "广度优先" — 覆盖生活工作全场景
  • "用户主导" — 用户决定Agent的行为
  • "透明可控" — Markdown文件，用户可编辑
  • "模型无关" — 支持任意模型接入

核心理念：让每个人都能拥有自己的AI Agent

辩证思考：这两种哲学并非对立，而是互补。Claude Code证明了AI Agent在专业领域的深度潜力，OpenClaw展示了AI Agent在通用场景的广度可能。未来理想的AI Agent可能需要兼具两者——既有Claude Code级别的专业深度，又有OpenClaw级别的通用广度。
